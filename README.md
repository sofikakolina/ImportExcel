# Excel Import — FastAPI + Next.js

Приложение для загрузки Excel-файла с организационными данными, их хранения в PostgreSQL, просмотра, редактирования и экспорта.

---

## Быстрый старт (Docker)

Самый простой способ запустить весь проект одной командой:

```bash
docker compose -p excel-import up --build
```

После запуска:
- Frontend: http://localhost:3000
- Backend API: http://localhost:8000
- Swagger UI: http://localhost:8000/docs

---

## 4. Как выполнить импорт файла

1. Откройте http://localhost:3000/import
2. Перетащите файл в зону загрузки или нажмите "Выберите"
3. Поддерживаемые форматы: `.xlsb`, `.xlsx`, `.xls`
4. Нажмите "Загрузить файл"
5. Прогресс импорта отображается в реальном времени через WebSocket
6. По завершении отображается результат: количество загруженных, пропущенных строк и возникшие предупреждения

**Логика повторного импорта:** при повторной загрузке того же файла система проверяет уникальность записи по комбинации `(employee_id, division_id, position_id, hire_date)`. Если запись уже существует — она обновляется (статус, зарплата, дата увольнения). Если записи нет — создаётся новая. Это позволяет безопасно переимпортировать файл без дублирования данных.

## 5. Архитектура данных

### Сущности и связи

```
departments (департаменты)
    └── divisions (отделы)  →  department_id → departments.id

employees (сотрудники — справочник ФИО)

positions (должности — справочник)

employments (записи о трудоустройстве — основная таблица)
    ├── employee_id   → employees.id
    ├── division_id   → divisions.id
    ├── position_id   → positions.id
    └── supervisor_id → employees.id (самоссылка)

operation_history (история операций импорта и экспорта)
```

### Описание таблиц

**`employees`** — справочник уникальных сотрудников по ФИО. Один сотрудник может иметь несколько записей в `employments` (разные периоды работы, разные должности).

**`departments`** — справочник департаментов.

**`divisions`** — справочник отделов, каждый принадлежит департаменту.

**`positions`** — справочник должностей.

**`employments`** — центральная таблица. Каждая строка — один период работы сотрудника в конкретном отделе на конкретной должности. Содержит: дату приёма, дату увольнения, статус (`Работает`/`Уволен`), тип занятости (`Штатный`/`Внештатный`), зарплату, руководителя и дату актуальности данных из файла.

**Уникальность:** `UNIQUE(employee_id, division_id, position_id, hire_date)` — не допускает дублей одного и того же периода работы.

**Актуальность на дату:** фильтрация через `hire_date <= дата AND (fire_date IS NULL OR fire_date >= дата)`.

### Почему такая структура

Данные из Excel представляют собой плоскую таблицу, где один сотрудник может встречаться несколько раз (уволен и принят повторно, разные должности). Нормализация в отдельные справочники позволяет избежать дублирования названий департаментов/отделов/должностей и строить эффективные запросы с фильтрацией. Разделение `employees` (справочник) и `employments` (записи) позволяет корректно отображать историю перемещений сотрудника.

## 6. Реализованные endpoints

### Импорт

| Метод | URL | Описание |
|---|---|---|
| `POST` | `/imports/` | Загрузка Excel-файла, запуск импорта |
| `GET` | `/imports/history` | История операций импорта |

### Сотрудники

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/employees/` | Список записей с фильтрацией и поиском |
| `GET` | `/employees/{id}` | Получение одной записи |
| `POST` | `/employees/` | Создание новой записи |
| `PATCH` | `/employees/{id}` | Редактирование записи |
| `DELETE` | `/employees/{id}` | Удаление записи |

**Параметры фильтрации для `GET /employees/`:**
- `search` — поиск по ФИО (частичное совпадение)
- `department_id` — фильтр по департаменту
- `division_id` — фильтр по отделу
- `status` — фильтр по статусу (`Работает` / `Уволен`)
- `employment_type` — фильтр по типу занятости
- `as_of_date` — фильтр актуальности на дату (формат `YYYY-MM-DD`)

### Справочники

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/departments/` | Список департаментов |
| `GET` | `/departments/divisions` | Список отделов (опционально `?department_id=`) |

### Экспорт

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/exports/` | Запуск экспорта |
| `GET` | `/exports/?tables=employees,departments&format=xlsx` | Экспорт выбранных разделов |
| `GET` | `/exports/download/{operation_id}` | Скачивание готового файла |
| `GET` | `/exports/history` | История операций экспорта |

### WebSocket

| URL | Описание |
|---|---|
| `WS /ws/{operation_id}` | Получение прогресса операции в реальном времени |

## 7. Как выполнить экспорт

1. Откройте http://localhost:3000/employees
2. Нажмите кнопку "Экспорт" в правом верхнем углу
3. Выберите разделы для экспорта (сотрудники, департаменты, отделы, должности)
4. Выберите формат: XLSX или CSV (zip-архив с отдельным файлом на каждый раздел)
5. Нажмите "Экспортировать"
6. Прогресс отображается через WebSocket, файл скачивается автоматически по завершении

Экспорт учитывает все изменения, внесённые через интерфейс после импорта.

## 8. Принятые решения и допущения

**Парсинг Excel:** файл читается через `pandas` с `engine='pyxlsb'` для `.xlsb` и стандартным движком для `.xlsx`. Первые 3 строки файла (заголовок и нумерация) пропускаются автоматически — ручное исправление файла не требуется.

**Даты в Excel serial формате:** числа вроде `45292` автоматически конвертируются в даты через формулу `date(1899, 12, 30) + timedelta(days=N)`. Строковые даты формата `dd.mm.yyyy` также обрабатываются.

**Опечатка "Дапартамент":** намеренно не исправляется — данные сохраняются как есть, без ручного вмешательства в исходный файл.

**Повторный импорт:** обновляет существующие записи (upsert по уникальному ключу), не создаёт дубли.

**Ошибки отдельных строк:** не прерывают весь импорт — строка пропускается, ошибка логируется и возвращается в результате операции.

**Экспорт в XLSB:** не реализован — стандартные Python-библиотеки не поддерживают запись в этот формат. Экспорт доступен в XLSX и CSV.

**Фоновые задачи:** импорт и экспорт выполняются через `BackgroundTasks` FastAPI с созданием новой сессии БД внутри задачи, чтобы избежать проблем с закрытием async-сессии.

**База данных:** PostgreSQL через async SQLAlchemy (`asyncpg`). Схема создаётся автоматически при старте через `Base.metadata.create_all`.

## 10. WebSocket: обмен статусами импорта и экспорта

**Endpoint:** `ws://localhost:8000/ws/{operation_id}`

Клиент подключается к WebSocket сразу после получения `operation_id` от HTTP-запроса на импорт или экспорт. Сервер отправляет сообщения в формате JSON:

```json
{
  "progress": 75,
  "status": "processing",
  "data": {
    "processed_rows": 15,
    "total_rows": 20
  }
}
```

**Возможные статусы:**
- `pending` — операция создана, ожидает запуска
- `processing` — выполняется
- `completed` — успешно завершена
- `completed_with_warnings` — завершена с предупреждениями
- `failed` — завершена с ошибкой

Соединение закрывается после получения финального статуса (`completed` / `failed`). При экспорте финальное сообщение содержит `file_url` для скачивания файла — фронтенд автоматически инициирует скачивание.

## 11. История операций

**Хранение:** все операции импорта и экспорта сохраняются в таблице `operation_history` в PostgreSQL. Таблица содержит: тип операции, статус, время запуска и завершения, имя файла или выбранные разделы, результат, ошибки и предупреждения, процент выполнения.

**Отображение:**
- История импортов — на странице `/import` внизу страницы, обновляется после каждой операции
- История экспортов — доступна через `GET /exports/history`

История сохраняется в БД и остаётся доступной после перезагрузки страницы и перезапуска фронтенда.

## Реализованные функции

- Загрузка и парсинг `.xlsb` / `.xlsx` файлов без ручного исправления
- Нормализованная структура БД с 5 связанными таблицами
- Полный CRUD для записей о трудоустройстве
- Фильтрация по ФИО, департаменту, отделу, статусу, типу занятости и дате актуальности
- WebSocket прогресс для импорта и экспорта
- История операций в БД
- Экспорт в XLSX и CSV (zip) с выбором разделов
- Автоматическое скачивание файла после экспорта
- Docker Compose для запуска всего стека одной командой

## Что не реализовано

- Экспорт в формат `.xlsb` (отсутствует поддержка записи в Python-библиотеках)
- Пагинация таблицы сотрудников
- Детальная страница сотрудника
- Сортировка по столбцам таблицы
